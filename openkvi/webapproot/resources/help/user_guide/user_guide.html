1. Introduction

OpenKVI is a web-application allowing dynamic control over KVM servers and Virtual Machines. It aims to allow system administrators to quickly perform common tasks.

If you haven't installed an OpenKVI VM along with the KVM server, you can create it with the following procedure:
1) Enable OpenKVI virtual machine: ncxctl kvm.openkvi --set enable
2) Set it's access bridge: ncxctl openkvi.access.bridge --set br_name
3) And it's access mode: ncxctl openkvi.access.mode --set [nat/direct] (nat is recommanded)
4) Copy the KVM iso file on the KVM server.
5) Then you can start OpenKVI creation with ncxctl openkvi.vm --create /path/to/kvm/iso

1.2 Interface Overview

The application is composed of three main parts:

  1. The Data Center panel on the left, that list all managed KVM nodes and Virtual Machines.
    The top most item in the list give access to OpenKVI global information and configuration; it will be referenced as the DataCenter further in this document.
  2. The Contextual Control panel on the right, that give control over the item selected in the left panel.
    Therefore available controls will differ depending on the nature (DataCenter, KVM node or VM) of the selected item.
  3. The Task List at the bottom, that display the list of ongoing and finished tasks.

Tool Bar ↦ View ↦ Switch to compact view
This mode removes all non-essential elements to maximize the space for important components, allowing for small screen users, such as laptops, to have a better user experience.

2.4.1 Network Configuration

The following procedure describes how to setup the node's low-level networks as well as Virtual Networks.
Low-level networking configuration consist in setting OpenVswitch bridges and Bonding requirements on the KVM node, while Virtual Networks are abstractions used for Virtual Machines configuration and which purpose is to hide the underlying complexity. It can also used to provide an homogenous ground for Virtual Machines across multiple KVM nodes thus easing VM's migration.

2.4.1.1 Node Networking

A node's networks must be configured directly on the KVM host, either on the serer's console screen or through a SSH connection (⚠ be carefull when configuring the network over a SSH connection, a wrong doing may cut your access).
The configuration is done with ncxctl command line tool.

Use ncxctl net.a.key --info to display information about net.a.key and how to use it.

How

1 Create Open vSwitch bridges.

KVM nodes' external access is handled by Open vSwitch stack, so the first step is to create those underplaying vswitches.
The recommendation is to create one vswitch per external access.
Use ncxctl net.bridges command for this task.

Eg. Creating ovsbr0, ovsbr1 and ovsbr2 bridges:

# ncxctl net.bridges --set ovsbr0 ovsbr1 ovsbr2

You can also use --add and --remove switches to add and remove Open vSwitch bridges.

2 External access: Physical interface.

Once OVS bridges are created you need to set their access interface.
This can be either a physical device like “eth0”or a bond.

Eg. Set eth0 as access interface to ovsbr0

# ncxctl net.conf.bridges.ovsbr0.access --set eth0

3 External access: Bonded interfaces.

To set a bond as an access interface you need to create it with ncxctl net.bonds command.

Eg. Bond eth2 and eth3 into bond2 interface:

# ncxctl net.bonds --set bond2:eth2,eth3

Then you can use this bond as an access interface.

Eg. Attach bond2 to ovsbr2:

# ncxctl net.conf.bridges.ovsbr2.access --set bond2

The following modes of bonding will work with any kind of upstream switch. On the upstream switch, do not configure the interfaces as a bond:

  • active-backup
    Assigns all flows to one slave, failing over to a backup slave when the active slave is disabled. This is the only bonding mode in which interfaces may be plugged into different upstream switches.
  • balance-slb
    Balances flows among slaves based on source MAC address and output VLAN, with periodic rebalancing as traffic patterns change.
The following mode requires the upstream switch to support 802.3ad with successful LACP negotiation and must be used in conjunction of the --lacp switch.
If LACP negotiation fails, then active-backup mode is used.
  • balance-tcp
    Balances flows among slaves based on L2, L3, and L4 protocol information such as destination MAC address, IP address, and TCP port.

4 Set Management IP addresses:

To set a management IP on a bridge use ncxctl net.conf.bridges.[br_name].mgnt.ip command.
This will create the [br_name]-mgnt interface on the system.

Eg. Set an IP address on ovsbr0:
# ncxctl net.conf.bridges.ovsbr2.mgnt.ip --set 192.168.0.1/24

It's also possible to set a VLAN id for this interface with ncxct net.conf.bridges.[br_name].mgnt.vlan command.

5 Restart the network.

Once all modifications have been done it is mandatory to restart the network.
However, due to the multiple network layers involved in virtual networks, it's not possible to simply use the system command (service network restart).
So to restart a KVM networking stack, use the following command:

# ncxctl services..networks --restart

2.4.1.2 Virtual Networks

The following procedure describes how to setup Virtual Networks.
Virtual Networks are abstractions based either on OpenVswitch (OVS) bridges or SR-IOV devices.
A third type exist, Private Network, which is a special OVS bridge without any external connection, but unlike both previous type, a Private Network does not need to be based on an existing device. Actually the underlying OVS bridge will be automatically created by OpenKVI and likewise destroyed once the Private Network is removed.

You can add VLAN tagging capabilities to your Virtual Networks, relieving the VMs the burden of tagging network packets and allowing for more flexibility.
Check Section 5 of the following procedure.

How

1 In the configuration tab select Networking.

Networking Configuration

2 Select the Virtual Networks tab.

3 To add a new Virtual Network, click on New at the bottom of the right panel.

4 In the pop-up window, in the General Settings section, set the Virtual Network's name (no blank allowed).

Select the Connection Type which can be either OpenVswitch, SR-IOV passthtrough (if available) or Private Network. Then, depending on the connection type, you will have to choose an existing OVS bridge or a SR-IOV device.

Add a Virtual Network


5 In the Portgroups section you can add one or more portgroups.
Portgroups provide support for multiple VLAN tagging at the Virtual Network level. Each Portgroup can handle at most one VLAN tag, but one Virtual Network can handle many portgroups.

To add a portgroup with VLAN tagging, click on the button at the bottom of the Portgroup list.
Set it's name, then toggle the Use VLAN tagging checkbox and in the Tag ID editor enter a number between 1 and 4094.

2.4.1.3 Virtual NIC migration

It’s possible to migrate a Virtual Machine’s Virtual NIC directly from the Virtual Networks configuration panel.

How

1 Just Drag and Drop Virtual Interfaces between Portgroups. It’s also possible to migrate a Virtual Interface between two portgroups of different Virtual Networks.

Virtual NIC migration

The Live migration of a Virtual Interface is only possible with Virtual Networks based on OpenVswitch.

3.5.2 Rollbacking a Snapshot

When rollbacking a snapshot, the Virtual Machine is reseted to the state it was before the snapshot.
The snapshot is then deleted and all modifications done within the snapshot are lost.

How

1 Select the required snapshot from the list

2 Click on the Rollback snapshot button (c.f. Figure below)

Rollback Snapshot


3.5.3 Saving a Snapshot

Saving a snapshot means that all modifications done in this snapshot will be merged with its parent, or in the main image if it doesn't have a parent snapshot, then the snapshot is discarded.
Thus at the end of the procedure any modification is saved and the snapshot removed.

How

1 Select the required snapshot from the list

2 Click on the Merge snapshot with parent button (c.f. Figure below)

Merge snapshot with parent


3.5.4 Advanced Snapshot usages

A more advanced usage of snapshots is also possible.

How

1 Reset a snapshot:
The Virtual Machine is reseted to the initial state it was in when creating the snapshot.
This give the possibility to erase any unwanted modifications but keeps the snapshot for further trials.

Reset Snapshot

2 Go to a snapshot:
When using multiple snapshots it is possible to move from one snapshot to another.
In the process, the Virtual Machine's data is restored to the initial state of the targeted snapshot.

Go to snapshot

3 Discard a snapshot:
When using multiple snapshots it is possible to discard an used snapshot.
If the snapshot has a child, then its state will be merged with its child. Otherwise any modification it might contains will be lost.

Discard snapshot

4.1 Roles

Three roles define the level of permission granted to a user.
Those roles are:

  • Adminstrator: has full control over OpenKVI.
  • PowerUser: has only a full control over Virtual Machines (Create/Destroy/Configure).
  • User: has a very limited control over Virtual Machines.
The role is assigned at user creation (c.f. 4.2 User Management) and can be modified by the administrator.

4.3 Authentication Modes

This procedure describes how to change OpenKVI authentication mode.
There is a choice between three different methodes, the default Local SQL database, against a Remote LDAP server or an Active Directory server.

How

1 Select Authentication in the left panel.

Authentication Settings

2 In the dropdown Methode field select the wished mode.

Authentication Modes

4.3.1 Local SQL database

This is the default authentication mode.
User's login and encrypted password are stored in a local database.
Each user must be manually added by the administrator (c.f. 4.2 User Management).

4.3.2 LDAP Authentication

In this mode, users's credential is checked against a remote LDAP server.
There is no needs for a LDAP Admin account as the LDAP connection is done with the user own credentials.

How

1 When selecting Remote LDAP server authentication, a new panel appears requiring at least five parameters to be filled:

  1. Host: the LDAP server IP address
  2. Port: the LDAP port (default is 389)
  3. Base DN: directory entry name
  4. Default Role: default role assigned to new users (c.f 4.1 Roles)
  5. Login: directory attribute holding the user's credentials
First name, Last name and Mail are optionals directory attributes that extend the user account information.

LDAP Parameters

2 Check On-the-fly user creation if you whish to automatically add new users to OpenKVI's database.
If unchecked, the administrator will have to add each user before they can log in.

3 Press Test LDAP connection to check that all parameters are correctly set.
A popup window will ask for a test user login/password, then you can proceed with the test.

4 If the above test is successful, then click on Save to validate the configuration.

OpenKVI is now configured to authenticate user against a remote LDAP service.

4.3.3 Active Directory Authentication

The Active Directory procedure is exactly the same as for a LDAP authentication setup (c.f. 4.3.2 LDAP Authentication), but with an additional mandatory parameter: the Domain name.

Active Directory Parameters

4.4 Security Level

Two Security Levels are available:

  1. The Low level allows unencrypted communications and there is no firewalling.
    This level should be mostly used for test purposes.
  2. The High level force encrypted communications, the firewall is active on OpenKVI as well as on every nodes' management IP.
    This level is suitable for a production environment.

How

1 Select Security in the left panel.

Security Settings

2 Slide the cursor on the wanted level.

Change Security Level

By default OpenKVI comes with its Security Level set to Low.
Only a member of the Administrator group can change the security level.
